### Changed

**Atomic iMessage consumers — adjudicated rows are rejected from corpus ingestion.** The owner-adjudication contract for `acquire_imessage_sent_atomic` is now explicit and enforced downstream: a row listed in `adjudicated-identity-exclusions.json` records the owner's rejection of that row from corpus ingestion, not an approval of its content. The sealed run tree still holds the row (completed-run bindings are immutable post-close), but `compose_imessage_review_bursts` validates the adjudication file against the atomic validator's identity-scan summary, drops every adjudicated row from burst content, treats it as a hard burst boundary, and accounts for it under the new `adjudicated_excluded_rows` / `adjudicated_excluded_words` conservation category in its checkpoint and receipt. Its source-config fingerprint binds the adjudication file digest (schema `setec-imessage-review-burst-source-config/2`, tool version 1.1), so a completed package refuses resume after any adjudication change. The supported direct `author_corpus_export` route now applies the same canonical validation and reject-from-corpus rule before reading row prose, and binds the adjudication digest into its opaque source-snapshot fingerprint.
