### Fixed

**`imessage_sent_atomic` — fixed a circular-import crash on direct script execution.** Running `acquire_imessage_sent_atomic.py` as a direct script (`python acquire_imessage_sent_atomic.py --validate-run ...` or any other CLI invocation) raised `ImportError: cannot import name 'WindowsPortableDurableRowIo' from partially initialized module 'windows_portable_tree'` on every call, because Python registers a directly-executed script under `sys.modules["__main__"]` rather than its own filename; `windows_portable_tree.py`'s module-scope `import acquire_imessage_sent_atomic as A` then found no entry under the canonical name and re-executed the file from scratch, colliding with `windows_portable_tree`'s own not-yet-finished module load. Importing the module normally (as the 6,800+-test suite and every other consumer does) already resolved the cycle via `sys.modules`, so this was invisible to CI while breaking every real CLI user. Fixed by aliasing `sys.modules["acquire_imessage_sent_atomic"]` to the running `"__main__"` module when executed directly, so the two identities collapse into the one real module object — the same resolution every non-`__main__` importer already gets, with no change to import timing or class identity for any other caller. Added a subprocess-level regression test (`--help` and a `--validate-run` against a missing path) so this class of defect — invisible under in-process import, live under direct script execution — can't go undetected again.
