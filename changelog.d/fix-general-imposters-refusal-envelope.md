### Fixed

**`general_imposters` hard refusals now emit a structured R3 refusal instead of crashing the R4 bounds gate.** When the harness refuses (fewer than `MIN_IMPOSTORS` distinct impostor personas, or no candidate docs) it set `proportion=float("nan")`; once `general_imposters` became a dispatcher-run contract surface, that NaN met `build_output`'s R4 output-validity gate and raised `OutputValidityError`, surfacing through `setec_run.py` as `internal_error` (exit 1) with a traceback (and crashing direct CLI use). A refusal now builds an `available: false` envelope via `build_error_output` with `reason_category: bad_input` (the manifest/corpus doesn't satisfy the method's preconditions); NaN never reaches the gate. Choosing `available: false` over a `null` proportion is deliberate — a held-out consumer scoring `null` would read a refusal as the worst-possible candidate, whereas "the validator could not run" must surface as blocking. The dispatcher now **honors a surface-emitted `available: false` envelope on both delivery paths** (stdout + file): it re-emits the surface's own `reason`/`reason_category` verbatim and derives the exit code from `reason_category` (`bad_input` → exit 3), so a script-emitted refusal and a dispatcher-synthesized one are indistinguishable to the consumer. An `available: false` envelope with a missing or unrecognized `reason_category` is a contract bug (a failure a consumer can't branch on); rather than pass the malformed envelope through, the dispatcher **synthesizes** a proper `internal_error` envelope in its place, so the consumer always gets a branchable `reason_category` (R3). The gray-zone *decision* (a valid proportion in [0.20, 0.80] where the harness declines an attribution *claim*) is unaffected — it ships `available: true` with a real number.
