### Added

**`acquire_blog` — `--bucket {impostor,test}` for routing acquired prose into the drift/test set, not just the impostor pool.** Every entry `acquire_blog.py` emitted previously hard-coded the impostor shape (`corpus_role: impostor`, `use: [voice_impostor]`, `split: baseline`), so acquiring your own AI-involved Substack into the *test/drift* bucket required a brittle one-off that monkey-patched `acquisition_core.compose_manifest_entry`. The bucket is now a first-class flag: `--bucket test` emits `use: [test_set]`, `split: test`, and **no** `corpus_role` (so the five impostor-only fields are omitted too), matching the hand-curated drift entries (e.g. `wordpress_archive_post2022_uncertain/`). Companion flags `--ai-status` (e.g. `mixed`) and `--notes-composite` (comma-separated authorship states → `notes.composite_states`, which the schema requires for `ai_status: mixed`) plus `--notes-description` cover the AI-involvement metadata. `--impostor-for` / `--consent-status` are now required only for `--bucket impostor` (enforced in `run()` before any network budget is spent), so the test bucket can omit them. **Behavior-preserving for the default:** with no `--bucket` the call passes the impostor presets and unchanged `ai_status` default, so `--help` aside, every impostor-bucket run emits a byte-identical manifest (covered by `test_substack_end_to_end` + a new `test_compose_impostor_shape_is_the_default` regression guard).
