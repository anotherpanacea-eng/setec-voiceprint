### Changed

**Deduplicated the LLM-judge provider plumbing into a shared `judge_backends.py`.** `narrative_judge` and `argument_judge` each carried their own near-identical `_api_judge_{anthropic,openai,gemini}` adapters (~220 lines of SDK import + client construction + request/response shape + `JudgeError` wrapping). That duplication is what let the #193 manifest/`_extract_json` hardening land in one family and not the other (the parity gap fixed in #194). The provider plumbing now lives once in `judge_backends.make_api_judge`, parameterized by each family's `build_user_content`, `build_result`, `JudgeError` class, and `extract_json` — so the families' **data contracts stay decoupled** (distinct `JudgeResult` shapes, prompts, parsing) while the plumbing can't drift again. No behavior change (identical request shapes, identical error messages, the provider call wrapped and text-extraction left unwrapped exactly as before); a regression test pins that both families resolve to the one shared `make_api_judge` object and that no per-family adapter remains. The `text_utils` and acquisition-argparse dedups in #198 remain a follow-up.
